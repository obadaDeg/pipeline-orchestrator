# Implementation Plan: Dashboard API Parity & Bug Fixes

**Branch**: `008-dashboard-api-parity` | **Date**: 2026-03-23 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/008-dashboard-api-parity/spec.md`

## Summary

Surface all existing backend capabilities in the dashboard and fix two broken pages. The backend already implements signing secrets, teams, pipeline editing, and user registration — none are accessible from the UI. Two dashboard pages crash today: `JobsPage` calls `GET /jobs` which does not exist, and `JobDetailPage` delivery-attempts pagination is broken because the backend returns no pagination metadata. This plan adds one new backend endpoint (`GET /jobs`), fixes one (`GET /jobs/:id/delivery-attempts`), adds a supporting `GET /teams` list endpoint, and implements five new or updated dashboard UI surfaces.

## Implementation Phases

### Phase 1 — Backend Bug Fixes (Highest Priority)

Fix the two broken endpoints first. These are independent of all UI work and immediately unblock the JobsPage and JobDetailPage.

**1a. Fix GET /jobs/:id/delivery-attempts pagination**
- Update `getDeliveryAttempts` in `jobs.controller.ts` to call `parsePagination(req.query)`.
- Update `job.service.ts` `getDeliveryAttempts()` to accept `{ limit, offset }` and return `{ items, total }`.
- Controller wraps with `paginatedResponse(items, total, page, limit)`.

**1b. Add GET /jobs endpoint**
- Add route in `jobs.router.ts` (before `GET /jobs/:id` to avoid path shadowing).
- Add `listJobsHandler` controller: parse pagination + optional `pipelineId` query param.
- Add `listJobs(userId, params)` service method: scoped to user's pipelines + team-owned pipelines the user is a member of.

### Phase 2 — Backend Supporting Endpoint

**2a. Add GET /teams list endpoint**
- Add route in `teams.router.ts`.
- Add `listTeamsHandler` controller.
- Add `listTeams(userId)` service: returns all teams where user is owner or member, including member count and `isOwner` flag.

### Phase 3 — Dashboard: Registration & Auth

**3a. RegisterPage**
- New `RegisterPage.tsx` with email + password form.
- Client-side password length validation (≥8 chars) before submit.
- On success: `auth.login(data.apiKey.key, data.user.email)`.
- Add `/register` route to `App.tsx`.
- Add "Create account" link to `LoginPage.tsx`.

### Phase 4 — Dashboard: Pipeline Enhancements

**4a. Pipeline Edit Mode**
- Add `isEditing` state, edit fields, Save/Cancel to `PipelineDetailPage.tsx`.
- On save: `PATCH /pipelines/:id`.

**4b. Signing Secrets Panel**
- New `SigningSecretPanel.tsx` component.
- Add "Security" tab to `PipelineDetailPage.tsx`.
- On mount: `GET /pipelines/:id/signing-secret`.
- Generate/Rotate: `POST /pipelines/:id/signing-secret` with ConfirmDialog for rotate.
- Revoke: `DELETE /pipelines/:id/signing-secret` with ConfirmDialog.

### Phase 5 — Dashboard: Teams UI

**5a. TeamsPage**
- New `TeamsPage.tsx`: lists teams from `GET /teams`. CreateTeam inline action.
- Add `/teams` route and sidebar link.

**5b. TeamDetailPage**
- New `TeamDetailPage.tsx`: members list from `GET /teams/:id`. Add/remove member actions.
- Add `/teams/:id` route.

### Phase 6 — Tests & MSW Handlers

**6a. Backend tests**
- Unit tests for `listJobs`, `getDeliveryAttempts` (pagination), `listTeams`.

**6b. Dashboard MSW handlers**
- Add all new handlers to `dashboard/src/test/handlers.ts` per `contracts/screens.md`.

**6c. Dashboard component tests**
- `RegisterPage.test.tsx`
- `SigningSecretPanel.test.tsx` (all four states)
- `TeamsPage.test.tsx`
- `TeamDetailPage.test.tsx`
- `PipelineDetailPage.test.tsx` (edit mode + security tab)

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| `GET /jobs` query slow at scale (joins across pipelines + team memberships) | Query uses existing indexed columns (`pipeline_id`, `owner_user_id`). Add `EXPLAIN ANALYZE` check during implementation. |
| Signing secret revealed in React state — could leak to error logs | Never log the `secret` field. Clear from state on unmount. |
| Team member lookup by email could expose user existence | Error message on "not found" should be generic: "No account found with that email." |
| Delivery attempts fix changes existing API response shape | Shape is additive only (new fields). Existing callers that read only `items` are unaffected. |
