# Implementation Plan: Delivery Attempts Tab & Per-Pipeline Rate Limiting

**Branch**: `011-delivery-attempts-tab` | **Date**: 2026-03-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/011-delivery-attempts-tab/spec.md`

## Summary

Two complementary features delivered together. The delivery attempts feature is largely already built (API, service, and dashboard UI exist); the only addition is `responseTimeMs` tracking throughout the delivery pipeline and its display in `JobDetailPage`. The rate limiting feature is entirely new: a `rateLimitPerMinute` column on `pipelines`, a Redis fixed-window counter service, integration into the webhook ingest path, and dashboard form fields.

**Scale/Scope**: Per-pipeline isolation; Redis key TTL handles automatic cleanup.

## Technology Decisions

### Rate Limit State Storage: Redis (ioredis, existing)

The project already has `ioredis` available via BullMQ (`src/queue/redis.ts` exports a standalone Redis client). Using it directly with `INCR`/`EXPIRE` adds zero new dependencies and two Redis round-trips per webhook request.

### Rate Limit Algorithm: Fixed-Window (60-second window)

Chosen for simplicity and predictability. The window boundary is the start of the current UTC minute. Key: `ratelimit:{sourceId}:{windowStartSec}` with TTL of 61 seconds. A sliding window would be more accurate but requires a sorted set and Lua scripting — unnecessary for protecting a job queue.

### Error Handling: RateLimitError extends AppError

Follows the existing error class pattern. The webhook controller catches it and adds the `Retry-After` header before forwarding the 422 response. The global error handler also handles it for any future path that might throw it.

### responseTimeMs: Measured in http-client.ts

`deliverPayload()` records `Date.now()` before and after the fetch call. On timeout or network error the value is `null`. This is the Delivery Engine boundary — keeping measurement there avoids coupling the delivery engine to timing logic.
